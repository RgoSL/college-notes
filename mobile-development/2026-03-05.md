# POO, Maven e Classes em Java

<br>

## 📌 POO — Conceitos Fundamentais

### Polimorfismo

- Múltiplas formas de um mesmo objeto  
- Permite comportamentos diferentes para o mesmo método  
- Subclasses podem redefinir comportamentos  
- Uso de sobrescrita (override)  
- Uso de sobrecarga (overload)  

### Abstração

- Foco nos aspectos essenciais do objeto  
- Oculta detalhes de implementação  
- Simplifica a modelagem do sistema  

### Contrato

- Define regras que devem ser seguidas  
- Estabelece comportamentos obrigatórios  
- Muito aplicado com interfaces  

<br>

## 📌 Maven

### Definição

- Ferramenta de gestão de projetos Java  
- Automatização de build  
- Baseada em POM (Project Object Model)  

### Arquivo POM

- Arquivo `pom.xml`  
- Define estrutura do projeto  
- Define fluxo de build  
- Gerencia dependências  

### Estrutura do Maven

- Versão do modelo  
- `groupId`  
- `artifactId`  
- Versão do projeto  
- Tipo de empacotamento  
- Nome da aplicação  
- URL do projeto  

### GroupId

- Identificador único da organização  
- Representa empresa ou desenvolvedor  

### ArtifactId

- Identificador do projeto  
- Nome do artefato  

### Empacotamento

- Define formato final do projeto  
- Exemplo: `jar`  

### Properties

- Configuração de versões (ex: Java)  

### Scope

- Define escopo das dependências  
- Exemplo: `test`  

<br>

## ⚙️ Funcionamento do Maven

- Projeto definido via `pom.xml`  
- Dependências são baixadas automaticamente  
- Build automatizado (compile, test, package)  
- Gerenciamento de versões centralizado  

<br>

## 💡 Observações

- Padroniza estrutura de projetos  
- Facilita gerenciamento de dependências  
- Muito usado com frameworks Java (ex: Spring)  

<br>

## ❗ Atenção

- Dependências mal definidas causam conflitos  
- Versões incompatíveis podem quebrar o build  
- Evitar excesso de dependências  

<br>

## 📌 Tipos de Classes em Java

### Classe Comum

- Declaração tradicional  
- Atributos definidos manualmente  
- Uso de getters e setters  
- Maior flexibilidade  

### Classe Record

- Estrutura simplificada  
- Definição direta dos atributos  
- Atributos imutáveis  
- Classe implicitamente `final`  
- Sem necessidade de getters/setters manuais  

<br>

## 💡 Observações

- Records reduzem código boilerplate  
- Ideal para DTOs e dados imutáveis  

<br>

## ❗ Atenção

- Records não permitem alteração de estado  
- Não substituem todas as classes tradicionais
